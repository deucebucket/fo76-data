## SeventySix.cdx - Cell Distance Index (804,536 bytes)

### Format Identification

The CDX is a **Cell Resource Index** file, directly referenced in Fallout76.ini:
```ini
SCellResourceIndexFileList = SeventySix.cdx
```

This is a binary spatial lookup table the engine uses to determine which BA2 archive resources to stream for any given cell position.

### Binary Structure

```
Header (8 bytes):
  [0x00] char[4]  magic    = "bcdx"
  [0x04] uint32   count    = 50,283 (total records)

Records (16 bytes each, 50,283 records):
  [+0x00] uint8   type         LOD/cell type code
  [+0x01] int8    cellX        signed cell X coordinate (-69 to +63)
  [+0x02] int16   cellY        signed cell Y coordinate (-1120 to +911), LE
  [+0x04] uint32  cellFormID   cell's Form ID from the ESM
  [+0x08] uint32  contentHash  hash of the cell's resource content
  [+0x0C] uint32  sourceHash   hash of the source archive/worldspace data
```

### Type Byte Encoding

The type byte encodes both the LOD level and whether the cell is exterior or interior:

| Type | Count | Description |
|------|-------|-------------|
| 0x05 | 16,247 | Exterior terrain LOD data |
| 0x07 | 16,924 | Exterior full-detail cell data |
| 0x08 | 16,247 | Exterior distant/previs LOD data |
| 0x85 | 226 | Interior cell (terrain LOD equivalent) |
| 0x87 | 413 | Interior cell (full detail) |
| 0x88 | 226 | Interior cell (distant LOD equivalent) |

Bit 7 (0x80) distinguishes interior cells (set) from exterior cells (clear). Interior cells have zero coordinates since they are not spatially positioned in the worldspace grid.

### Cell Grid Structure

- **9 exterior worldspace cells** (persistent cell IDs), with the dominant one being **0x0025DA15** (FormID 2,480,661) -- this is the **Appalachian Wasteland** worldspace
  - 0x0025DA15 accounts for 48,622 of the 50,283 records (96.7%)
  - The remaining 8 exterior cells (0x00002ED3, 0x00004B16, 0x00635F96, etc.) are likely DLC worldspaces (The Pitt, Nuclear Winter arena, etc.)
- **436 interior cells** -- dungeons, vaults, buildings, instances
- Cell X ranges from -69 to +63 (roughly 133 cells wide)
- Cell Y ranges from -1120 to +911 (but in 16-cell grid blocks)

### Grid Block Pattern

Y coordinates follow a strict 16-cell block pattern, only appearing at block boundaries:
- Block start values: 0, 16, 32, 48, 64, ...
- Block end values: 15, 31, 47, 63, 79, ...

This indicates the CDX indexes at **grid block granularity** (16x16 cell blocks), not individual cells. Each block boundary entry maps to the content hash needed to stream that block's resources.

### Content Hashing

- 18,693 unique content hashes across 50,283 records
- Multiple grid positions can share the same content hash (identical terrain/LOD tiles)
- The dominant source hash **0x54E4A503** appears in 17,337 records -- this is likely the hash for the base SeventySix.esm archive
- 32,947 unique source hashes total, indicating many records point to update archives

### How the Game Uses This

1. Player moves to grid position (X, Y)
2. Engine looks up the CDX for that position across all three LOD types (0x05, 0x07, 0x08)
3. Content hash identifies which specific resource block to load from the BA2 archives
4. Source hash identifies which archive (base ESM or which update BA2) contains the data
5. Engine streams the appropriate terrain, meshes, and LOD data

This is critical for FO76's streaming architecture -- the game never loads the entire map, only the blocks near the player. The CDX is essentially a **spatial hash table** for the streaming system.

## SeventySix.bin - Integrity Token (137 bytes)

### Structure

137 bytes of high-entropy binary data with no ASCII content and no recognizable magic bytes.

```
Byte layout:
  [0x00-0x08]   9-byte header
  [0x09-0x88]   128-byte payload (1024 bits)
```

### Analysis

**Not a simple hash:** The file does not contain the ESM's MD5, SHA-1, SHA-256, or CRC32. No known hash of SeventySix.esm or SeventySix.cdx appears anywhere in this file.

**Header (9 bytes):**
- Byte 0: `0x40` (64) -- possibly a version or format identifier
- Bytes 1-4: `0x0DD8EFFF` -- no match to ESM header fields
- Bytes 5-8: `0xFFE82765` -- high entropy, no obvious structure

**Payload (128 bytes = 1024 bits):**
The 128-byte payload is exactly the size of an **RSA-1024 digital signature** or two concatenated SHA-512 hashes. Given that FO76 is an online game with server-side content validation:

**Most likely interpretation: Server Content Verification Token**

This file is almost certainly a **cryptographic signature** generated by Bethesda's servers, used to verify the ESM/data files have not been tampered with. Evidence:
1. 137 bytes is not a standard hash size but matches a compact signature format (header + RSA-1024 sig)
2. High entropy throughout (105 unique byte values out of 137 bytes) -- consistent with encrypted/signed data
3. No correlation to any local file hash -- it is signed server-side, not computed locally
4. FO76 requires online authentication; this token validates the client's data files against the server's expected state
5. File timestamp (March 17, 2026) is more recent than the CDX (March 7), suggesting it was regenerated during the last client validation/patch

**Alternative interpretation:** It could be a **combined hash digest** -- 9-byte header (version + flags) plus two 64-byte SHA-512 hashes of different file components (ESM header hash + ESM content hash). This would allow the client to quickly verify file integrity without rehashing the entire 920MB ESM.

## Cryptid Hunting Relevance

### CDX File
The CDX reveals the **exact grid block structure** of the game world. For cryptid hunting:
- We can map which grid blocks exist and which are empty/void
- Content hash changes between blocks indicate unique content vs. repeated tiles
- The 9 exterior worldspaces confirm the number of distinct playable areas
- Grid block boundaries at 16-cell intervals define the streaming unit size
- Cells with unique content hashes (not shared with neighbors) are more likely to contain unique encounters or hidden areas

### BIN File
The integrity token means Bethesda can detect client-side file modifications. This is relevant because:
- Any ESM edits for exploration would trigger server-side validation
- The token is regenerated per patch, confirming it is tied to specific file versions
- Client-side content discovery must use observation/memory reading rather than file modification

### INI Insights
- Ultra LOD at 30x means the engine can render objects at extreme distances -- cryptid spawn points at the edge of render distance would only be visible on Ultra
- The missing CSG geometry file is curious -- was combined static geometry removed? This could relate to how the engine handles hidden/inaccessible areas
- NW.esm still being referenced means Nuclear Winter map data may still be accessible in memory
